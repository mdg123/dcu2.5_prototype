## 1. 현황 요약 (1차 탐색 결과)

### 현재 구조
- **메인 UI**: `public/lrs/index.html` — 7개 하위 메뉴 (홈/학습현황/학업성취/학습수행/맞춤학습/교사활용지수/일일현황)
- **백엔드 API**: `routes/lrs.js` — 18개 엔드포인트
- **DB 스키마**: `learning_logs` (원본) + 5개 집계 테이블 (`lrs_daily_stats`, `lrs_user_summary`, `lrs_content_summary`, `lrs_class_summary`, `lrs_service_stats`)
- **수집 지점**: 채움클래스(수업/과제/평가), 콘텐츠(조회), 출석, 커뮤니티, Ingest API(외부)

### 핵심 문제
1. **스키마 불일치**: 저장 로직(15개 컬럼)과 테이블 정의(10개 컬럼) 미스매치 → `statement_json`, `achievement_code` 누락
2. **성취기준 미연동**: curriculum의 achievement_code가 learning_logs에 저장 안 됨 → 교과별/성취수준별 분석 불가
3. **공백 UI**: Empty state 과다 (실데이터 거의 없음)
4. **기간 필터 버그**: from/to와 days 드롭다운 충돌
5. **내보내기 미연동**: `/api/lrs/export` 있으나 버튼 없음
6. **AI 인사이트 미구현**: class-analytics 구조만 있음
7. **모바일 반응형 부족**

## 3. 단계별 실행 계획

### Phase 1 — 병렬 분석 (1차 라운드)
- A. **LRS 전문가**: 현재 데이터 모델 감사 + xAPI 호환 리모델링 제안 + KPI 30개 나열
- B. **UI 디자이너**: 7개 메뉴 IA 재설계 + 와이어프레임 + 시각화 라이브러리 선정
- C. **정책기획자**: 교육부/사업 요구사항 관점에서 필수 지표 정의 + 학생/교사/관리자 롤별 KPI

→ **PM 통합**: 세 전문가 의견 통합 → 확정 사양서 작성

### Phase 2 — 설계 확정
- 데이터 스키마 업데이트 (statement_json, achievement_code 추가)
- API 엔드포인트 사양 확정 (신규/개선 포함)
- UI 컴포넌트 명세 확정
- 더미 시드 데이터 생성 스크립트 설계 (realistic fake data)

### Phase 3 — 병렬 구현
- **DB/백엔드 팀**: 스키마 마이그레이션, 집계 함수, API 엔드포인트 개선
- **각 서비스 팀**: logLearningActivity 호출 보강 + metadata 풍부화
- **UI 팀**: 7개 메뉴 프론트엔드 리빌드 + 신규 컴포넌트
- **시드 데이터 팀**: 실사용 시나리오 기반 더미 데이터 생성

### Phase 4 — 통합 검토 라운드 1
- **테스터**: 모든 메뉴 시나리오 테스트, 에러/버그 목록
- **LRS 전문가**: 데이터 정합성 감사, 집계 로직 검증
- **UI 디자이너**: 시각 품질 감리, 반응형 체크, 접근성
- **감리**: 코드 품질, 보안, 성능

→ **PM 취합**: 이슈 트래커 작성, 심각도 분류

### Phase 5 — 수정 사이클
- 발견된 이슈 우선순위대로 수정
- 재검토 → 재수정 루프 (최대 3회전)

### Phase 6 — 최종 검수
- 전체 통합 테스트
- 스크린샷/영상 캡처
- 커밋 + 푸시

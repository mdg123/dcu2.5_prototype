### 2.5 반응형 브레이크포인트

설계서 §3.6은 **1920 / 1280 / 768 / 360** 4단을 명시.
구현은 `@media (max-width:1024px/768px/480px)` 3단 (L248~263).

- **1280 breakpoint 누락**: 현재 1024 이상은 전부 KPI 4열. 13~14인치 노트북의 1280 구간에서 사이드바 220px + KPI 4열이 빡빡해질 수 있다. 1280 이하에서 콘텐츠 패딩 축소 또는 KPI 3열 전환 분기 권장.
- **360px 대신 480px 사용**: 일반적 모바일 하한(iPhone SE 320~375)은 커버되지만 설계서가 요구한 "요일만 축약 바차트"(§3.6)·"테이블 카드 변환"(§4.3)은 트리거되지 않는다.
- **768px에서 사이드바 아이콘 전용**은 설계서 §3.6과 일치 (L253).

### 2.8 기간 픽커 단일화 (B4 버그 해결)

`dc-period-picker`가 단일 컴포넌트로 "최근 7일 / 30일 / 90일 / 사용자 지정" 4개 칩 + 커스텀 날짜 인라인 펼침(L46~57) 구조. `customRange` 요소는 `hidden` 토글(L567)되며 적용 버튼으로만 쿼리 갱신(L572) — **B4의 "드롭다운 + from/to 이중 상태" 충돌이 구조적으로 불가능**한 설계. 통과.

다만 설계서 §4.11은 "커스텀 모드에서는 좌우 두 달력을 나란히 표시"를 명시했는데 구현은 `<input type="date">` 2개만 사용. 브라우저 네이티브 date picker이므로 기능상 문제 없으나, 설계서에서 약속한 경험은 아니다. 의도적 간소화로 보이며 감점 사유는 아님.

### 2.10 색대비 (WCAG AA)

`dc-tokens.css`의 주요 조합을 sRGB 기준으로 계산:

| 전경 | 배경 | 대비율 | 판정 |
|------|------|--------|------|
| `--gray-900 #111827` on `#fff` | — | 16.89:1 | AAA |
| `--gray-700 #374151` on `#fff` | — | 9.73:1 | AAA |
| `--gray-500 #6b7280` on `#fff` | 본문 14px | 4.83:1 | **AA 통과** |
| `--gray-400 #9ca3af` on `#fff` | sync-time/subtitle | 2.85:1 | **AA 실패 (일반 텍스트)**, 비필수 보조문자라 AA Large도 경계치 |
| `--lrs-primary #2563eb` on `#fff` | 브레드크럼·링크 | 5.17:1 | AA |

**주요 이슈**:
1. `sync-time` / `subtitle` / `state-panel` 본문이 모두 `--gray-500` 또는 `--gray-400`. L84(`sync-time`)는 `--gray-400`을 사용 — **명확한 AA 실패**. `--gray-500` 또는 `--gray-600`으로 상향 권장.
2. 히트맵의 `data-h="2"` 셀(중간 밀도)의 흰 숫자는 2.51:1로 **AA 불통**. 이 셀에 한해 숫자 색을 `--gray-800`으로 전환하거나 텍스트를 제거하는 수정 필요 (현재 css L174가 다른 어두운 셀과 묶여 `color:#fff`).

### 2.11 픽셀 단위 / 정렬 이슈

1. **L55의 "적용" 버튼 인라인 스타일**: `<button style="background:var(--lrs-primary);color:#fff;...">적용</button>` — 디자인 토큰화된 컴포넌트 안에 **인라인 스타일이 섞여** 있어 `dc-period-picker .chip[aria-pressed="true"]`와 시각적 충돌(둘 다 같은 primary 배경이 되어 "커스텀" 칩과 "적용" 버튼이 구분되지 않음). 전용 `.dc-period-picker .apply` 클래스 권장.
2. **L189 KPI 카드 `role="group"`**: 설계서 §4.1은 `role="link"` + tabindex=0 + Enter/Space 핸들러. 현재 role/핸들러 불일치 — 스크린리더가 "그룹"이라고만 읽어 "누를 수 있는 카드"임을 전달하지 못함.
3. **L225, L227 KPI value 32px vs 28px**: 설계서 §3.3은 학생 뷰 KPI를 32px로 키워 "성취감"을 주라고 명시. 현 구현은 전 역할 동일 `--fs-display:28px`. 학생 뷰에서 `.dc-kpi-card--student { font-size:32px }` 추가 권장.
4. **`dc-chart-wrapper .chart-body { height:320px }` 고정** (L144): 두 개 차트를 `grid-2`로 나란히 놓으면 모두 320px 고정. 설계서 §3.1의 히트맵은 정사각형 셀이 이상적이라 별도 높이 제어 필요. 현재는 히트맵 자체가 렌더되지 않아 드러나지 않음.
5. **학생 Hero 카드 없음**: 설계서 §3.3의 gradient primary→purple Hero 카드가 `s-home`에 전혀 없다. KPI 4장 바로 노출. 학생 랜딩의 감정적 진입점이 빠져 "화이팅!" 체험이 사라졌다.
6. **교사 우리반 히트맵(요일×교시)** (설계서 §3.1) — 완전 누락. 현재 `t-home`는 "일별 활동량" 막대로 대체되어 **2축 밀도 정보가 1축으로 축소**되었다.
7. **경고 카드 `border-radius` 계산 오류 가능성**: L207 `::before { border-radius: var(--radius-md) 0 0 var(--radius-md) }`는 top-left/bottom-left만 10px. 카드 자체 radius와 맞물려 좋음. 단 `width:4px`인 요소에 10px radius를 주면 실제 렌더는 4×4 곡률로 클리핑되어 거의 둥글지 않아 보임 — 의도와 일치하면 OK이나 `radius-sm(6px)` 정도가 더 자연스럽다.
8. **사이드바 220px vs 설계 200px** (L225): 설계서 §3(ASCII) 기준 200px → 실구현 220px. 20px 차이는 시각적으로는 미미하나 1280 기준 콘텐츠 영역이 20px 좁아진다. 큰 문제는 아님.
9. **`dc-kpi-grid` 모바일 2열(1fr 1fr) L261 vs 설계 "KPI 2열×2행"**: 일치. 양호.
10. **`dc-drawer` 너비 420 / 모바일 90vw**: 설계서 §4.7의 "모바일에서는 바텀시트(전체 높이 90%)"는 미구현. 현재는 가로 90vw 드로어로만 표시되어 작은 화면에서 레이아웃이 비좁다.

## 4. 설계서 대비 잘 된 지점

짚어둘 만한 성과:
1. **토큰 네임스페이스 분리**(`--lrs-*` vs 기존 `--primary`) — `common-nav.css`와 충돌 없이 LRS 전용 시스템을 얹은 설계 판단이 적절. 주석(L3~6)으로 근거까지 명시.
2. **IA 매핑**의 수학적 정확성 — 4/5/4 메뉴 구성과 역할별 권한 행렬이 설계서와 1픽셀 오차 없이 일치.
3. **B4 버그 해결**이 단순 수정이 아니라 구조 개편 — `period + from/to`를 한 컴포넌트 내부 상태로 캡슐화.
4. **`prefers-reduced-motion` 대응**이 토큰 레벨에서 처리되어 컴포넌트마다 분기할 필요가 없음 — 설계서 §6.6을 가장 우아하게 해석.
5. **라우팅이 해시 기반**이라 페이지 전환 시 사이드바 깜빡임이 없고(§6.1), URL 공유가 가능(교사 ↔ 학생 ↔ 관리자 딥링크 시연 가능).

## 5. 최종 의견

Phase 3 구현은 **설계서의 "뼈대와 원칙"을 충실히 반영**했고, 실제 화면 5개 중 4개(교사 우리반 / 학생 학습요약 / 관리자 전체통계 / 관리자 데이터품질)가 정상 렌더된다. **접근성 3건(gray-400 대비, 히트맵 중간셀, 드로어 포커스 트랩)과 액션 딥링크 목업 1건만 Phase 3.1에서 해결하면 베타 테스트 투입이 가능**하다고 판단한다.

누락된 시각 컴포넌트(히트맵·스파크라인·Hero·실시간 피드)는 "있으면 좋은" 영역이 아니라 설계서에서 **랜딩 경험의 핵심**으로 지정된 요소들이므로, Phase 4 범위로 넘기기 전에 최소 골격이라도 구현할 것을 권장한다.

**승인 조건**: 본 보고서의 P0 3건을 다음 주 배포 전까지 수정하고, P1 5건은 Phase 3.1 범위로 별도 스프린트를 편성한다.
